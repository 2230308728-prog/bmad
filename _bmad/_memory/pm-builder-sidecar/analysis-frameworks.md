# PM Builder Analysis Frameworks

## 需求分析方法论

### 1. 五个为什么 (5 Whys)

**目的**: 挖掘真实需求，识别根本原因

**步骤**:
1. 陈述需求
2. 问"为什么需要这个？"
3. 重复4次，每次都问"为什么"
4. 最终答案通常是真实需求

**示例**:
- 需求: "我们需要一个AI聊天功能"
- 为什么？→ "因为用户想要快速得到答案"
- 为什么？→ "因为现在的搜索太慢了"
- 为什么？→ "因为搜索结果不相关"
- 为什么？→ "因为我们不知道用户真正在找什么"
- 真实需求: **改进搜索相关性，而不是添加AI聊天**

---

### 2. 用户旅程映射

**目的**: 确保流程完整性，不遗漏关键环节

**步骤**:
1. 识别用户角色
2. 定义用户目标
3. 列出所有接触点
4. 绘制完整流程
5. 识别遗漏步骤

**关键问题**:
- 用户从哪里开始？
- 每个步骤的触发条件是什么？
- 有哪些分支路径？
- 流程如何结束？

---

### 3. 影响-可行性矩阵

**目的**: 优先级评估，聚焦核心价值

**评估维度**:
- **业务影响**: High / Medium / Low
- **技术可行性**: Easy / Medium / Hard

**优先级规则**:
1. **High + Easy** → 立即做
2. **High + Medium** → 尽早做
3. **High + Hard** → 规划做
4. **Medium + Easy** → 填空做
5. **Medium + Medium/Hard** → 评估后决定
6. **Low + 任何** → 不做或延后

---

### 4. 真伪需求检查清单

**伪需求信号**:
- ❌ 来自竞争对手的功能
- ❌ 技术团队想玩的新技术
- ❌ 高层领导的一句话需求
- ❌ 没有明确用户群体的功能
- ❌ 无法量化的价值

**真需求验证**:
- ✅ 有明确的用户痛点
- ✅ 有清晰的使用场景
- ✅ 可以量化业务价值
- ✅ 有数据支持决策
- ✅ 团队有资源实现

---

## 快速参考

### 分析流程

```
1. 理解背景 (Context)
   ↓
2. 分析需求 (Analyze)
   ↓
3. 梳理流程 (Flow)
   ↓
4. 评估优先级 (Prioritize)
   ↓
5. 提供建议 (Recommend)
```

### 关键问题

每次分析时问：
- **为什么？** (5次)
- **用户是谁？**
- **场景是什么？**
- **价值有多大？**
- **资源够吗？**
